# 已有需求注释表达重构 Skill

## 目标

本 Skill 用于对已经接入完成的需求注释进行存量表达优化。

当前项目通常已经完成：

1. `src/requirements` 需求注册表已生成；
2. 页面数字角标已接入；
3. 点击角标可查看悬浮需求面板；
4. 右侧 PRD 阅读面板已接入；
5. 点击右侧需求卡片可定位页面锚点并高亮页面对象。

本 Skill 不负责重新生成业务需求，也不负责重新实现页面角标、悬浮需求面板或右侧 PRD 面板。

本 Skill 只负责：

1. 清洗已有需求注释表达；
2. 去除 `display` 和 `operation` 之间的重复描述；
3. 修正“后续流程”“状态反馈”“使用范围”“异常边界”等维度归属不准确的问题；
4. 删除用户侧无信息量兜底说明；
5. 修正多层编号导致的阅读问题；
6. 保证悬浮需求面板和右侧 PRD 阅读面板继续正常读取清洗后的内容；
7. 如果发现 Skill3 / Skill4 中存在硬编码注释内容，改为统一读取 registry。

处理后的注释应接近产品经理在 Axure 原型中写需求注释的表达方式：

- 显示说明：写清页面在不同状态、不同条件、不同数据边界下怎么展示；
- 操作说明：写清用户能做什么、什么时候能做、操作后系统怎么响应、异常时怎么处理。

## 当前执行阶段

本 Skill 适用于 Skill4 已经执行完成之后。

也就是说，本次处理的是**存量内容优化**，不是前置流程。

不要要求用户重新运行：

```txt
requirement-registry-writer
requirement-marker-reviewer
prd-prototype-integrator
```

除非发现现有文件缺失、结构损坏，或者右侧 PRD 面板并没有读取 `src/requirements` 中的 registry。

## 执行边界

### 允许修改

允许修改：

```txt
src/requirements/<页面或流程>.registry.ts
src/components/prd/requirement-utils.ts
src/components/prd/RequirementFloatingCard.tsx
src/components/prd/RequirementPanel.tsx
```

其中，修改组件文件仅限于：

1. 确保悬浮需求面板读取清洗后的 registry 内容；
2. 确保右侧 PRD 面板读取清洗后的 registry 内容；
3. 过滤空兜底说明；
4. 避免渲染层再次扩写、重复生成栏目；
5. 移除硬编码注释内容；
6. 修正重复编号渲染问题。

### 不允许修改

不允许修改：

```txt
业务页面主体逻辑
页面视觉布局
页面交互流程
需求编号 id
anchorId
anchorStatus
activate 路径
sourceType
页面角标位置
右侧 PRD 面板开关逻辑
拖拽宽度逻辑
高亮定位逻辑
```

不允许因为表达优化而删除业务事实。  
不允许因为两条需求看起来相似，就擅自合并需求编号。  
不允许重新生成全部 registry。  
不允许把已确认业务逻辑改成新的产品方案。  
不允许维护 registry 和组件硬编码两份注释内容。

## 必须读取的文件

执行前必须读取：

```txt
src/requirements/index.ts
src/requirements/schema.ts
.ai/skills/shared/logic-writing-spec.md
```

如果用户指定页面或流程，必须读取对应文件：

```txt
src/requirements/<页面或流程>.registry.ts
```

同时必须检查以下文件是否存在：

```txt
src/components/prd/RequirementFloatingCard.tsx
src/components/prd/RequirementPanel.tsx
src/components/prd/requirement-utils.ts
```

如果文件不存在，不要强行创建新面板。  
本 Skill 不负责重新接入 Skill3 / Skill4，只能在已有实现基础上做注释表达治理。

## 必须先检查内容来源

在清洗 registry 之前，必须先确认悬浮需求面板和右侧 PRD 面板当前展示内容的来源。

重点检查：

1. 悬浮需求面板是否从 registry 读取 `display` 和 `operation`；
2. 右侧 PRD 面板是否从 registry 读取 `display` 和 `operation`；
3. 是否存在重复维护的硬编码需求文案；
4. 渲染层是否会自动补齐“页面展示 / 状态反馈 / 操作规则 / 后续流程 / 异常边界”等栏目；
5. 渲染层是否会把空兜底说明展示给用户；
6. 数组项是否已经带有 `1、`、`2、`，同时渲染层又额外编号，导致多层编号。

处理规则：

### 情况 A：展示内容统一来自 registry

如果悬浮需求面板和右侧 PRD 面板都统一读取：

```txt
src/requirements/<页面或流程>.registry.ts
```

则只清洗 registry 中的 `display` 和 `operation` 文案。

不要修改页面代码。  
不要重做右侧 PRD 面板。  
不要重新接入角标。

### 情况 B：组件中存在硬编码注释内容

如果发现下面文件中复制了需求注释内容：

```txt
src/components/prd/RequirementFloatingCard.tsx
src/components/prd/RequirementPanel.tsx
业务页面组件文件
```

必须删除硬编码注释，改为从 registry 读取。

处理原则：

1. registry 是唯一需求注释来源；
2. 悬浮需求面板只负责展示 registry；
3. 右侧 PRD 面板只负责展示 registry；
4. 不允许维护两份注释内容；
5. 不允许改完 registry 后，页面仍然展示旧硬编码文案。

### 情况 C：渲染层会二次扩写栏目

如果组件渲染时会自动补齐下列栏目：

```txt
页面展示
状态反馈
操作规则
使用范围
后续流程
异常边界
```

需要取消二次扩写。

渲染层只负责展示 registry 中已有的结构，不负责重新生成注释内容。

如果 registry 清洗后没有某个栏目，渲染层不要强行展示空栏目。

### 情况 D：渲染层展示空兜底说明

如果组件会展示下列内容：

```txt
无额外权限限制
无额外数据流转
无异常场景
本对象无操作入口
本对象仅展示
沿用页面权限
暂无
无
空字符串
空数组
```

需要在 `requirement-utils.ts` 或展示组件中增加过滤逻辑，避免这些内容出现在用户侧悬浮面板和右侧 PRD 面板中。

## 清洗总原则

### 1. 业务事实不减少

清洗表达时，必须保证原注释中的有效业务事实仍然保留。

有效业务事实包括：

- 页面对象是什么；
- 默认展示什么；
- 不同状态下展示什么；
- 什么条件下展示、隐藏、置灰、禁用；
- 用户能做什么；
- 用户不能做什么；
- 点击、拖拽、上传、删除、确认、取消后的系统响应；
- 操作后的状态变化；
- 数据来源；
- 数据去向；
- 结果保存位置；
- 下游流转；
- 异常、失败、空状态、未识别、未匹配等边界；
- 已确认的产品决策；
- 明确的限制规则。

不得删除这些事实，只能重新组织表达。

尤其不得删除以下关键边界：

```txt
不得推测题号
未识别到题号时显示「未识别到题号」
识别中不可操作
取消识别不清空已框选区域
识别失败时保留原框和原有题号信息
答案文件不默认自动切题
答案文件未手动画框时由全局答案匹配承接
裁剪还原恢复到原始图片，而不是最近一次裁剪前图片
```

如果原 registry 中存在这些规则，清洗后必须保留。

### 2. 同一业务事实只写一次

同一个业务事实不要在多个维度中重复写。

例如，原始重复表达：

```txt
显示说明：
点击开始识别后，框变为识别中，并且不可操作。

操作说明：
点击开始识别后，本次参与识别的框继续显示，变为识别中，不可勾选、拖动、缩放或删除。
```

清洗后应拆分为：

```txt
显示说明：
识别中状态下，参与识别的框继续显示在资料页上，状态标签显示为「识别中」，框体呈不可操作样式。

操作说明：
点击「开始识别」后，系统只处理当前选中的框；识别过程中，这些框不可勾选、移动、缩放或删除。
```

规则：

- 显示说明可以写“看起来是什么状态”；
- 操作说明可以写“用户做了什么、系统怎么响应”；
- 操作说明中可以引用状态名，但不要重复完整视觉描述；
- 显示说明中不要写完整操作链路。

### 3. display 只写展示逻辑

`display` 只负责回答：

```txt
用户在页面上能看到什么？
不同状态下分别看到什么？
什么条件下展示、隐藏、置灰或变更文案？
页面展示的数据边界是什么？
```

可以包含：

- 页面结构；
- 默认展示；
- 可见文案；
- 标签；
- 按钮可见性；
- 状态文案；
- 条件展示；
- 禁用态；
- 空状态；
- 加载态；
- 成功态；
- 失败态；
- 数据统计展示；
- 数据缺失时的展示方式。

不要包含：

- 用户怎么操作；
- 点击后系统怎么处理；
- 操作结果传给哪里；
- 权限说明；
- 后续流程；
- 技术实现细节；
- “帮助用户理解”“便于用户操作”这类解释性废话。

### 4. operation 只写操作逻辑

`operation` 只负责回答：

```txt
用户能不能操作？
在哪里操作？
什么条件下能操作？
操作后系统怎么响应？
异常时怎么处理？
结果保存或流转到哪里？
```

可以包含：

- 操作入口；
- 可操作条件；
- 不可操作条件；
- 点击规则；
- 拖拽规则；
- 上传规则；
- 删除规则；
- 确认规则；
- 取消规则；
- 还原规则；
- 操作后的状态变化；
- 操作后的数据更新；
- 操作失败后的处理；
- 取消后的保留逻辑；
- 结果保存和下游流转。

不要包含：

- 页面默认长什么样；
- 状态标签的完整视觉描述；
- 与操作无关的页面展示；
- 大段重复显示说明；
- 解释性废话；
- 技术实现语言。

### 5. 按真实涉及的维度写，不强行补齐栏目

不要为了显得完整而每条需求都写满：

```txt
页面展示
状态反馈
操作规则
使用范围
后续流程
异常边界
```

只有当前需求真的涉及该维度，才保留该栏目。

推荐栏目如下。

#### display 下可用栏目

```txt
页面展示
状态与条件
数据边界
```

使用规则：

- `页面展示`：写默认能看到什么、对象结构、主要文案；
- `状态与条件`：写不同状态、不同模式、不同条件下的展示差异；
- `数据边界`：写页面展示数据的统计口径、缺省值、未获取时怎么展示。

如果某条需求只有一种固定展示，不要强行拆出 `状态与条件`。  
如果不涉及统计口径或数据缺失，不要强行写 `数据边界`。

#### operation 下可用栏目

```txt
操作规则
结果与流转
异常边界
使用范围
```

使用规则：

- `操作规则`：写用户怎么操作、什么时候可操作、操作后页面如何响应；
- `结果与流转`：只在确实涉及数据保存、进入下游、传递给下一步时使用；
- `异常边界`：只写失败、缺失、取消、不可用、未匹配、未识别等非正常场景；
- `使用范围`：只在该需求存在特殊权限、特殊角色、特殊范围限制时使用。

如果权限只是“沿用页面权限”，不要在用户侧注释里展示。  
如果没有异常，不要强行写“无异常场景”。  
如果没有下游数据流转，不要强行写“无额外数据流转”。

### 6. 默认不使用“后续流程”

除非原始需求真的描述了下一步流程，否则不要使用“后续流程”作为小标题。

很多情况下，“后续流程”其实是下面几类内容之一：

| 原标题 | 实际含义 | 应改成 |
|---|---|---|
| 后续流程 | 数据从哪里来 | 数据边界 |
| 后续流程 | 数据传到哪里 | 结果与流转 |
| 后续流程 | 点击后状态变化 | 操作规则 |
| 后续流程 | 下一步页面怎么展示 | 状态与条件 |
| 后续流程 | 加入试卷、提交、导入 | 结果与流转 |

示例：

原始表达：

```txt
后续流程：
匹配统计数据来源于已识别的题目、答案和解析数据。
```

清洗后：

```txt
数据边界：
匹配统计根据当前已识别的题目、答案和解析数据计算。
```

原始表达：

```txt
后续流程：
裁剪后的图片数据保存在题目对象中，加入试卷时传递给下游。
```

清洗后：

```txt
结果与流转：
确认裁剪后，裁剪结果保存到当前题目；加入试卷时，系统使用裁剪后的图片。
```

### 7. 删除用户侧无信息量兜底说明

用户侧注释中不要展示下列内容：

```txt
无额外权限限制
无额外数据流转
无异常场景
本对象无操作入口
本对象仅展示
沿用页面权限
暂无
无
空字符串
空数组
```

如果 schema 需要这些字段，可以保留为内部结构化字段，但悬浮需求面板和右侧 PRD 面板不得展示。

### 8. 纯展示对象只保留极短操作说明

如果某个对象是纯展示，不需要强行扩写操作说明。

可以写：

```txt
该提示条不可点击、不可收起、不可展开。
```

不要写：

```txt
帮助用户了解当前状态。
方便用户后续操作。
提示用户进行下一步处理。
```

纯展示对象的操作说明应尽量短。

### 9. 操作类对象可以少写显示说明

如果某个需求重点是按钮、入口、拖拽、删除、裁剪、确认、取消等操作，可以只在显示说明中写清入口位置和状态，不要重复写完整业务链路。

示例：

```txt
显示说明：
编辑模式下，题目卡片展示「使用图片」开关；开启后展示题目图片和裁剪入口。

操作说明：
用户开启「使用图片」后，系统切换为图片展示；点击裁剪入口进入裁剪模式，确认后保存裁剪结果，取消后退出裁剪模式且不保存。
```

### 10. 避免多层编号

清洗后用户看到的内容只保留一层编号。

不要出现：

```txt
1、
页面展示
1、
1、内容
```

应整理为：

```txt
页面展示
1、内容
2、内容
3、内容
```

如果 registry 中的字段本身是数组，数组项不要再内嵌 `1、`、`2、` 前缀，避免渲染层重复编号。

如果渲染层已经负责编号，registry 文案中不要手写编号。  
如果 registry 文案已经手写编号，渲染层不要再次编号。  
优先推荐：registry 中不手写编号，由渲染层统一展示编号。

### 11. 避免技术实现表达

用户侧注释不要出现下列技术表达：

```txt
state
props
callback
component
父组件
子组件
接口名
字段名
mock
sessionStorage
className
DOM
数组下标
```

如果原始注册表中有技术表达，应转成产品语言。

例如：

```txt
裁剪结果写入 question.imageCrop 字段
```

应改为：

```txt
裁剪结果保存到当前题目。
```

除非用户明确要求展示技术字段，否则不要把技术实现细节写进悬浮需求面板或右侧 PRD 面板。

## 清洗步骤

### 第 1 步：确认目标 registry

如果用户指定页面或流程，直接处理对应 registry。

如果用户没有指定页面或流程：

1. 读取 `src/requirements/index.ts`；
2. 列出当前已有 registry；
3. 如果只有一个 registry，直接处理；
4. 如果有多个 registry，先询问用户要处理哪一个；
5. 不要一次性清洗所有 registry，除非用户明确要求。

### 第 2 步：检查 Skill3 / Skill4 的读取方式

检查：

```txt
RequirementFloatingCard
RequirementPanel
requirement-utils
```

判断展示内容是否来自 registry。

如果来自 registry：

```txt
只清洗 registry。
```

如果存在硬编码：

```txt
移除硬编码，改为读取 registry。
```

如果存在渲染层二次扩写：

```txt
取消二次扩写，改为按 registry 原结构展示。
```

如果存在空兜底展示：

```txt
增加过滤逻辑，不在用户侧展示空兜底说明。
```

### 第 3 步：逐条识别重复信息

对每条 requirement 检查：

1. `display.description` 与 `operation.description` 是否重复；
2. `页面展示` 与 `状态反馈` 是否重复；
3. `操作规则` 与 `后续流程` 是否重复；
4. `异常边界` 是否重复正常操作规则；
5. `使用范围` 是否只是无信息量兜底；
6. 是否存在同一事实换句话说多次出现；
7. 是否出现多层编号；
8. 是否出现标题和内容不匹配，例如“后续流程”实际在讲数据来源；
9. 是否有技术实现表达；
10. 是否有解释性废话；
11. 是否遗漏关键边界。

### 第 4 步：按正确维度重新归类

把每条有效事实重新归入对应位置：

```txt
页面怎么展示 -> display / 页面展示
不同状态怎么展示 -> display / 状态与条件
展示数据怎么算 -> display / 数据边界
用户怎么操作 -> operation / 操作规则
操作结果保存或传递 -> operation / 结果与流转
失败、取消、缺失、不可用 -> operation / 异常边界
特殊角色或权限限制 -> operation / 使用范围
```

如果某个事实同时涉及展示和操作，拆成两句话：

- 展示侧只写结果状态；
- 操作侧只写触发规则和响应逻辑。

### 第 5 步：重写为用户可读表达

重写后的句子应满足：

1. 句子短；
2. 一条只讲一个事实；
3. 不堆叠多个逗号；
4. 不重复解释目的；
5. 不使用技术实现语言；
6. 不写成开发说明；
7. 不写成测试用例；
8. 不写成营销文案；
9. 不出现空泛表达。

推荐表达：

```txt
页面展示：
1、步骤3左侧资料预览区按页展示上传资料。
2、每页右上角展示当前页已框选数量。
3、题目+答案分文件模式下，资料页先展示题目文件，再展示答案文件；文件标签根据上一步指定的文件用途显示为「题目」或「答案」。
```

不推荐表达：

```txt
该区域用于帮助用户了解当前资料处理情况，方便用户后续进行题目识别和答案匹配操作。
```

### 第 6 步：验证展示不会重复

修改完成后，必须确认：

1. 悬浮面板不会展示空兜底说明；
2. 右侧 PRD 面板不会展示空兜底说明；
3. 渲染层不会自动补齐不存在的栏目；
4. 数组项不会被重复编号；
5. 清洗后的 `display` / `operation` 能被现有组件正常读取；
6. 页面角标仍能正常点击；
7. 右侧 PRD 卡片仍能正常展示；
8. 点击右侧 PRD 卡片仍能定位锚点；
9. 高亮和 activate 路径没有被破坏。

## 疑似重复需求处理

如果发现两个或多个 requirement 描述的是同一页面对象、同一操作、同一状态，只是拆分方式不同，不要直接合并。

应在完成摘要中列出：

```txt
疑似重复需求：
- REQ-001 与 REQ-006 都在描述「开始识别后框选区状态变化」
- 建议后续确认是否合并为一个需求，或保留一个作为页面展示、一个作为操作规则
```

默认模式下，不做合并。

只有用户明确要求合并时，才可以执行需求合并。

合并时必须：

1. 保留较早的需求编号；
2. 不复用被删除编号；
3. 更新相关 anchorId；
4. 更新 Skill3 / Skill4 依赖的引用；
5. 在摘要中说明合并前后变化。


如果项目没有 `ts-check`，查看 `package.json` 中可用脚本，选择最接近的 TypeScript 校验命令。

如果只改 registry 文案，通常不需要运行 dev server。

如果同时修改了 `RequirementFloatingCard`、`RequirementPanel` 或 `requirement-utils`，需要检查：

1. 页面可以正常打开；
2. 数字角标仍可点击；

6. 高亮仍有效；
7. 空兜底说明不展示；
