# 页面逻辑审核 Skill

## 目标

从产品设计者和测试者两个视角审核当前前端页面。不要直接改页面，也不要直接生成最终 PRD。核心目标是发现页面里还没有明确的业务逻辑、边界场景、异常处理和交互规则，并把这些问题整理成用户可以快速决策的选项。

## 必须遵循的流程

1. 先判断用户是否已经说明本次要审核的页面或流程范围。
2. 再判断用户是否已经说明页面相关前提信息，例如目标用户、业务背景、当前页面目标、已知约束、这次希望重点审核的方向。
3. 如果缺少页面名称或页面范围，不要开始审核，先询问用户要审核哪个页面、路由、文件或业务流程。
4. 如果缺少页面前提信息，不要开始审核，先询问用户补充页面背景或业务场景；如果用户没有额外前提，可以让用户回复“无额外前提”。
5. 当页面范围和前提信息都齐全后，用一句话复述本次审核范围和前提，再开始读取代码。
6. 读取代码时，先看 `package.json`，再看相关路由页面、导入组件、弹窗、hooks、本地状态、props、API 调用、缓存/存储、页面跳转和恢复逻辑。
7. 审核范围内每个有业务含义的 UI 对象，包括：
   - 字段
   - 文案
   - 按钮
   - 操作区域
   - 弹窗/面板
   - Tab/步骤条
   - 列表/卡片/表格区域
   - 空状态/加载态/成功态/失败态/禁用态
8. 审核时同时扮演两个角色：
   - 产品设计者：判断业务场景、用户意图、页面文案、流程路径、决策点是否清晰。
   - 测试者：判断边界场景、权限、数据流转、异常状态、禁用状态、重复操作、状态恢复是否覆盖。
9. 只输出会影响业务逻辑、PRD 编写、测试验收或后续实现的待确认问题。纯视觉偏好不要提，除非它会影响用户理解或操作结果。
10. 首次审核结果必须使用“首次输出格式”。
11. 用户回复类似 `1A2C3D` 后，解析每个编号选择。如果用户对某些编号输入了自定义文字方案，按“自定义方案处理规则”记录；如果用户对某些编号表达没看懂或有疑问，按“疑问回复处理规则”处理。
12. 只有当本轮所有待确认问题都已经有明确决策后，才使用“决策确认格式”统一回显完整决策结果。
13. 用户确认决策结果无误后，才把确认结果写入 `docs/prd-workflow/decisions/<页面或流程>.decision.md`。

## 前置信息检查

开始审核前必须检查两类信息：

1. 页面名称或页面范围  
   示例：`/homework`、上传录题弹窗、作业管理页、试卷编辑页、AI 小乐识别资料流程。

2. 页面前提信息  
   示例：目标用户是谁、页面要解决什么业务问题、当前版本哪些能力已确定、哪些能力只是原型占位、这次重点关注哪些逻辑。

如果缺少信息，按下面方式提问：

```md
我需要先确认两项信息，再开始审核：

1. 页面名称/页面范围：<如果已知就写已知内容；未知就请用户补充>
2. 页面前提信息：<如果已知就写已知内容；未知就请用户补充，允许用户回复“无额外前提”>
```

不要在缺少必要信息时直接开始审核。

## 首次输出格式

首次审核结果必须用中文，并严格使用下面的一级结构：

```md
一、页面名称
<页面名称或流程名称>（<路由或主要文件路径>）

二、待确认问题
**1、<字段/文案/按钮/某区域>：<对象名称>**

**问题描述：**<说明当前代码已实现什么、还缺少什么业务情景、为什么需要确认。>

**建议方案：**
A. <方案A，写清采用后的业务含义>
B. <方案B，写清采用后的业务含义>
C. <方案C，写清采用后的业务含义>

---

**2、<字段/文案/按钮/某区域>：<对象名称>**

**问题描述：**...

**建议方案：**
A. ...
B. ...
C. ...

你可以直接回复类似：1A2C3B
```

首次输出规则：

- 编号从 `1` 开始连续递增。
- 每个问题只能对应一个字段、文案、按钮、区域或逻辑对象。
- 编号对象行必须加粗，例如：`**1、按钮：确认按钮**`。
- 问题描述标题必须写成 `**问题描述：**`，不要写成“关于【对象】的问题描述”。
- 建议方案标题必须写成 `**建议方案：**`。
- 编号对象行、问题描述、建议方案之间各空一行。
- 每个编号问题之间用 `---` 分割；如果问题很少，也可以用两行空行分隔，但优先用 `---`。
- 每个问题建议提供 2-4 个互斥方案。只有确实需要时才提供更多。
- 同一个编号下的所有建议方案必须回答同一个决策维度，方案之间必须可比较、可互斥；不要把“支持范围”“实现方式”“入口策略”“异常处理口径”等不同维度混在同一组选项里。如果发现多个维度都需要确认，必须拆成多个编号问题。
- 建议方案必须完整到可以直接沉淀为 PRD 或开发需求。凡是方案里包含提示、报错、警告、空状态、失败状态、禁用状态、二次确认、拦截或恢复动作，必须同时写清触发条件和具体提示文案，不要只写“给提示”“显示错误”“弹窗确认”等笼统表述。
- 如果提示文案本身还不确定，必须把文案作为该编号下的可选方案，或拆成独立编号让用户确认。
- 只有当“不处理/保持现状”本身是合理产品决策时，才加入这个选项。
- 不要输出冗长背景、代码逐行讲解或实现计划。
- 如果问题太多，优先输出 P0/P1 问题，并说明页面可以分批审核。
- 不要把多个无关决策混在同一个编号问题里。

## 用户决策回复解析

当用户回复类似 `1A2C3D` 时，按下面规则解析：

- `1A` 表示第 1 个问题选择 A 方案。
- `2C` 表示第 2 个问题选择 C 方案。
- `3D` 表示第 3 个问题选择 D 方案。

用户回复可能不是纯编号选项，必须支持下面情况：

1. 标准选项  
   示例：`1A2C3D`。按编号和选项字母记录。

2. 自定义方案  
   示例：`1A2C3【改成上传前先让用户选择资料类型，再进入导入弹窗】5B`。  
   对第 3 个问题记录为“自定义方案”，并完整保留用户输入的文字，不要强行改写成已有 A/B/C/D。

3. 表示没看懂或提出疑问  
   示例：`1A2C3D4【我没看懂这个问题，是不是指关闭弹窗后数据会丢失？】5A7D`。  
   对第 4 个问题不要记录决策，按“疑问回复处理规则”先解释，等用户补充第 4 个问题的决策后，再统一回显完整决策结果。

4. 混合回复  
   同一条回复里可以同时包含标准选项、自定义方案和疑问。必须先记录已明确的决策，保留未决问题，不能丢失用户已经选择的其他编号。

如果用户回复里有无效编号、无效选项或无法判断的内容，只询问需要修正的那几项，不要重新输出全部审核内容。

## 自定义方案处理规则

当用户没有选择 A/B/C/D，而是对某个编号输入自己的解决方案时：

- 将该编号的决策标记为“自定义方案”。
- 原样保留用户输入的核心方案文字。
- 可以在决策确认时做轻微结构化整理，但不得改变用户方案含义。
- 写入决策记录时，`决策` 写为“自定义方案”，`结论` 写用户自定义方案的完整业务含义。
- 如果用户自定义方案里同时包含疑问或条件，先确认歧义，不要直接写入最终决策。

自定义方案在决策确认格式中这样展示：

```md
**3、按钮：确认按钮**

**决策结果：**自定义方案，<用户输入方案的完整业务含义。>
```

## 疑问回复处理规则

当用户对某些编号表示没看懂、有疑问、反问含义、要求解释时：

1. 先暂存其他已经明确的编号决策。
2. 不要输出完整决策结果。
3. 只针对用户有疑问的编号进行扩展说明。
4. 扩展说明后，重新给出该编号可决策的方案；可以沿用原 A/B/C/D，也可以根据用户疑问补充更清晰的新方案。
5. 明确告诉用户只需要继续回复未决编号，例如：`请继续回复：4A`。
6. 用户补齐未决编号后，再把之前暂存的明确决策和新补齐的决策合并，使用“决策确认格式”统一输出完整决策结果。

疑问回复格式如下：

```md
一、已记录决策
已先记录：1A、2C、3D、5A、7D。

二、需补充说明的问题
**4、<字段/文案/按钮/某区域>：<对象名称>**

**扩展说明：**<用更直白的话解释这个问题在问什么、为什么会影响业务逻辑或测试验收。>

**重新提供建议方案：**
A. <方案A>
B. <方案B>
C. <方案C>

你可以继续回复：4A
```

## 决策确认格式

解析用户选择后，必须用中文回显，并严格使用下面结构：

```md
一、页面名称
<页面名称或流程名称>（<路由或主要文件路径>）

二、决策结果
**1、<字段/文案/按钮/某区域>：<对象名称>**

**决策结果：**<选择的方案字母>，<该方案的完整业务含义。>

---

**2、<字段/文案/按钮/某区域>：<对象名称>**

**决策结果：**<选择的方案字母>，<该方案的完整业务含义。>

请确认以上决策是否正确。确认后我会把它沉淀为页面决策记录。
```

在用户确认前，不要写入决策记录文件。

## 决策记录文件

用户确认决策结果后，创建或更新：

```txt
docs/prd-workflow/decisions/<页面或流程>.decision.md
```

文件结构如下：

```md
# <页面名称或流程名称> 决策记录

## 页面范围
- 页面/流程：<名称>
- 路由：<route>
- 相关代码文件：
  - <file>

## 已确认决策
1. <字段/文案/按钮/某区域>：<对象名称>
   - 决策：<选择的方案字母>
   - 结论：<完整业务含义>
   - 影响范围：<页面显示、操作、权限、数据流转、异常处理等>

## 暂不处理
- <如有>

## 待后续确认
- <如有>
```

决策记录是后续“需求注册表” skill 的输入材料。文件必须简洁、具体、可追溯，不要写入未确认建议。

## 审核关注点

输出待确认问题前，至少检查下面维度：

- 入口：谁能看到入口、什么时候可点击、关闭后回到哪里。
- 文案：按钮文案、空状态文案、提示文案、状态文案是否和实际行为一致。
- 状态：加载、空、成功、失败、禁用、部分成功、恢复、重复操作状态是否完整。
- 权限：不同角色、数据归属、受限操作是否需要明确规则。
- 数据流转：数据从哪里来、保存到哪里、跨页面跳转/刷新/关闭弹窗后如何处理。
- 异常：非法输入、缺少输入、接口失败、请求超时、重复提交、中断流程、不支持的文件或数据。
- 隐藏交互：弹窗、悬浮面板、Tab、步骤、抽屉、Popover、hover 后才出现的操作。
- 校验：当前代码是否阻止了不可能或高风险操作。
- 恢复：用户是否能取消、重试、恢复、重新上传、继续流程，并避免数据丢失。

## 输出约束

- 面向用户的输出必须以决策为中心。
- 不要把所有小观察都塞给用户。
- 使用页面上的具体对象名称。
- 不要提出“是否优化体验”这类空泛问题。
- 每个选项都必须能成为后续 PRD 或实现决策。
- 对提示、报错、拦截、禁用、空状态、失败状态、确认弹窗等异常或状态处理，必须输出可落地的文案和触发条件；没有具体文案的方案不算完整方案。
- 除文件路径、代码标识、字段名、命令名等必要内容外，尽量使用中文。
